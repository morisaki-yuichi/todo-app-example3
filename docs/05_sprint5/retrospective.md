## Keep（続けること）

- **雛形はそのまま1コミット → 手を入れる差分を分離**。写経者が「どこからが
  自分で書くコードか」を差分で判別できる
- **フロントの API 呼び出しを `/api` プレフィックスに統一**。プロキシ・
  将来の接続先変更・grep のしやすさ、すべてに効く
- **「予想と違った」実験結果を差分として記録する**（実験⑥の 404 予想 → 実際は
  200 + HTML）。予想が外れたときこそ理解が進む

## Problem（困ったこと・うまくいかなかったこと）

- テンプレート生成物の削除（App.css / assets）を Step 5-3 のコミットに含めたが、
  「テンプレート掃除」と「画面実装」が1コミットに混ざり、差分がやや読みにくい。
  掃除だけ先に別コミットにすればよかった
- Home のテストで「ローディング表示」を検証する際、fetch モックの設計
  （永遠に解決しない Promise）に一工夫が必要だった。非同期 UI のテストは
  「どの時点の画面を検証しているか」を常に意識する必要がある

## Try（次のスプリントで試すこと）

1. **T-9: 「掃除コミット」を分ける**（テンプレート・不要コードの削除は、
   機能実装と別コミットにする）
2. **T-10: API 呼び出しを専用モジュール（src/api/）に集約してから画面を作る**
   （S6 は fetch が増える。画面に fetch を直書きすると、S9 の TanStack Query 移行で
   全画面を触ることになる。呼び出し口を1枚かませておく）

→ この Try を S6 で実際に使い、結果を S6 のレトロに記録する。

- フロント（5176）と API（8002）はポート違いで「別オリジン」。開発中は Vite プロキシで
  同一オリジンに見せるので CORS 設定が要らない（外すと JSON パースエラーという
  意外な顔で現れる）
- SPA は「HTML 1枚 + JS が画面を組み立てる」方式。未知のパスに index.html を返す
  フォールバックがあるため、API パスのタイポは 404 にならないことがある
- 「読み込み中 / 成功 / 失敗」は判別可能なユニオン型で1つの状態にする。
  boolean を並べると「ありえない組み合わせ」が型の上で作れてしまう
- package-lock.json は uv.lock の npm 版。CI は npm ci で「ロックと完全一致」を強制する
