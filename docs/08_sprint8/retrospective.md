## Keep（続けること）

- **「壊れている状態を先に観察してから直す」**（実験⑨ → CORS 導入 → 同じコマンドで
  成功を確認）。導入前後の差分が具体的な数字（405 → 200）で残る
- **api 層集約（S6 T-10）の威力**: 認証方式の切り替えという大工事で、
  画面コンポーネントは1行も変わらなかった。層を切る判断の成功例として教材化
- **対比表による記録**（cookie vs JWT）。「どちらが良いか」でなく
  「何と何の交換か」を表にする形式は、技術選定の記録として再利用できる

## Problem（困ったこと・うまくいかなかったこと）

- Step 8-2〜8-4 の間、アプリ全体としては動かない過渡状態があった（計画どおりだが、
  写経者が途中で止めると「壊れた状態」で終わる）。トレースガイドに注意書きを
  入れたが、横断変更の粒度設計はもっと上手になれる余地がある
- Node 25 の実験的機能との干渉という「教材の想定外」に遭遇した。解決はしたが、
  写経者の環境（Node のバージョン）によって再現したりしなかったりする問題は
  教材泣かせ（トラブル記録に環境条件を明記して対処）

<summary>回答例・ヒント（クリックで展開）</summary>

- CORS エラーはサーバのエラーではない。サーバは応答を返しており、許可ヘッダーの
  無い応答をブラウザが JS に渡さないだけ（だから curl では再現できない）
- JWT は「署名つきの読める札」。暗号化ではないので秘密を入れない。サーバに状態が
  無いぶんスケールするが、即時失効できない——cookie セッションと利点・欠点が
  ちょうど裏返しになっている
- プリフライトは「Authorization ヘッダーつきの越境リクエストを送ってよいか」の
  お伺い（OPTIONS）。CORS ミドルウェアはこれに答える口を作る
- 認証方式の総取り替えでも、fetch を api 層に集約してあれば画面は無傷。
  層の切り方は「将来どんな変更が来るか」への賭けであり、今回は賭けに勝った
