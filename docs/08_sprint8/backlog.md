## スプリントゴール

**「フロントと API が別オリジンのまま（プロキシなしで）通信できる。
認証は cookie セッションから JWT（Bearer トークン）に移行し、
両方式の違い（保管場所・失効・CSRF/XSS）が教材として記録されている」状態を作る。**

## なぜ移行するのか（このスプリントの意味）

cookie + プロキシ構成は開発では快適だが、本番でフロント（CDN 配信）と API
（別ドメイン）を分けるSPA 構成では「別オリジン」問題を避けられない。
このスプリントで**同一オリジンの魔法を捨て**、ブラウザの同一オリジンポリシーと
CORS を正面から扱う。認証もヘッダー方式（JWT Bearer）に切り替え、
cookie 方式との得失を対比する。

## 設計上の主な選択（理由つき）

- **JWT は HS256 署名・有効期限60分・ペイロードは user_id（sub）のみ**:
  最小構成で仕組みを見せる。リフレッシュトークンはスコープ外（概念解説で言及、
  実装は発展課題）
- **フロントの保管場所は localStorage + Authorization ヘッダー**:
  httpOnly cookie 方式（XSS に強いが CSRF 対策と CORS の credentials 設定が必要）
  との対比を概念解説で明記する。localStorage は XSS に弱い——このリスクを
  隠さず教材にする
- **logout エンドポイントは廃止**: JWT はサーバ側に消すべき状態がない
  （= **即時失効できない**という JWT の本質的な限界）。ログアウトは
  クライアントがトークンを捨てるだけ。cookie セッション（行削除で即失効）との
  最大の対比ポイント
- **sessions テーブルは削除マイグレーション**: 使わないテーブルを残すと
  「これは何？」という負債になる。downgrade で復元可能にする
- **CORS の許可オリジンは .env 駆動**（`FRONTEND_ORIGIN`）: `*` にしない。
  「どこからのブラウザ実行を許すか」は明示管理する

## スコープ外（やらないこと）

- リフレッシュトークン・トークンの自動更新（発展課題）
- トークンのブラックリスト等による JWT 失効機構（概念解説で言及のみ）
- TanStack Query（S9）
