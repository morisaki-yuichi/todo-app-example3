## 2. 開発前の質疑応答（2026-07-05）

### Q1. TODO 一覧のデフォルト仕様（ソート順・1ページの件数）

**決定**: 作成日時の降順（新しいものが上）・1ページ 10 件・offset 方式のページネーション。

- ソートは `created_at DESC` を既定とする（もっとも素直で説明コストが低い）
- ページネーションはクエリパラメータ `page`（1 始まり）と `per_page`（既定 10・上限 100）
- 絞り込みはクエリパラメータで指定: `completed=true|false`（状態）、`q=キーワード`
  （title / description の部分一致）
- 一覧レスポンスには総件数（`total`）を含め、フロントのページャ表示に使えるようにする

### Q2. 認証の段階移行と、フロント/バックの結合方針

**決定**: cookie セッション + Vite プロキシ → JWT + CORS へ移行し、両者を対比する。

- 第1部〜第2部前半: cookie セッション認証。フロントは Vite の開発プロキシ経由で API を
  呼ぶ（ブラウザから見ると同一オリジンになるため CORS 設定が不要）
- 第2部後半: JWT + CORS に移行。プロキシを外して「CORS エラーが出る」失敗実験を挟み、
  同一オリジンポリシーを体験的に学べる構成にする
- cookie vs JWT（保管場所・CSRF・スケール特性）の対比は概念解説集で扱う

### Q3. 写経者が 1 スプリントに割ける学習時間

**決定**: 3〜4 時間を上限の目安とする。

- 実装が重いスプリントは 3〜4h に収まるよう分割する（ベロシティ調整の基準）

### Q4. GitHub リポジトリの公開設定とブランチ戦略

**決定**: パブリックリポジトリ・GitHub Flow。

- リポジトリはパブリックで作成（教材からコミット / PR へのリンクを誰でも参照できるように）
- ブランチ戦略は GitHub Flow: `main` + 作業ブランチ + Pull Request。
  CI（GitHub Actions）が緑になってからマージし、マージ後に `main` で動作確認まで行う
- コミットメッセージは Conventional Commits（`feat:` `fix:` `docs:` `test:` `ci:` 等）

## 3. 確定した仕様サマリ

### TODO の項目定義（姉妹プロジェクト準拠・固定）

| 項目 | 型 | 制約 |
|---|---|---|
| id | 整数（自動採番） | 主キー |
| title | 文字列 | **必須**・最大 100 文字 |
| description | 文字列 | 任意・最大 1000 文字 |
| due_date | 日付 | 任意・過去日も許可 |
| completed | 真偽値 | 既定 `false` |
| user_id | 整数 | 所有者（外部キー）。他人の TODO への操作は 403 |
| created_at / updated_at | 日時 | 自動設定 |

### 認証・認可

- ユーザー登録・ログイン・ログアウトを実装する
- パスワードはハッシュ化して保存（bcrypt）
- 各ユーザーは自分の TODO だけを閲覧・操作できる。他人の TODO への操作は **403**、
  未ログインは **401**

## 4. 開発途中の仕様変更（追記欄）

（変更が発生したら、日付つきでここに追記する）

### 2026-07-05: ユーザーストーリー作成時に確定した詳細仕様

- **認証の資格情報**: メールアドレス + パスワードとする。パスワードは最小8文字。
  重複メールアドレスの登録は `409 Conflict`
- **ログイン失敗時のメッセージ**: メールアドレスの存在有無が推測できないよう、
  「メールが存在しない」「パスワードが違う」を区別しない同一メッセージで `401` を返す
- **編集 API**: `PATCH` による部分更新とする（指定した項目だけを更新）
- **実装順序の設計判断**: 認証（S4）より先に TODO CRUD（S2〜S3）を作るため、
  S2〜S3 の TODO は所有者なしで実装し、S4 で users テーブルと `user_id` を
  **既存データのマイグレーションつきで** 追加する。「稼働中テーブルの変更は
  データ移行つきで」を体験する教材上の意図的な設計（詳細は user-stories.md）

### 2026-07-05: スプリント4 で確定した詳細仕様・技術選定

- **パスワードハッシュは bcrypt ライブラリを直接使う**（当初案は passlib/bcrypt）。
  passlib は長期未メンテで bcrypt 4.x 以降との組み合わせ問題が知られているため。
  学習概念（ソルト・遅いハッシュ）は変わらない
- **cookie セッションはサーバサイド保存（sessions テーブル）**: httpOnly +
  SameSite=Lax。有効期限7日。ログアウト = セッション行の削除（即時無効化）
- **登録（POST /auth/register）は 201 を返しつつ自動ログイン**（第2部のフロント実装で
  「登録 → 即一覧画面」の動線にするため）
- **既存 TODO の引き取り先**: マイグレーションで `legacy@example.com`
  （ログイン不可のパスワードハッシュ `!`）を作成して割り当てる
- **スコープ外と明記**: パスワードリセット・メール確認・レート制限・CSRF トークンの
  本格実装（概念解説では扱う）

### 2026-07-05: スプリント8（JWT + CORS 移行）で確定した仕様

- **JWT は HS256・有効期限60分・ペイロードは sub（user_id）のみ**。
  リフレッシュトークンはスコープ外（発展課題）
- **フロントの保管場所は localStorage + Authorization: Bearer ヘッダー**。
  httpOnly cookie 方式との得失（XSS に弱い / CSRF に強い / CORS 設定が単純）は
  概念解説集の対比表に記録。リスクは隠さず教材化する
- **logout エンドポイントは廃止**（JWT に即時失効はない。ログアウトは
  クライアントのトークン破棄のみ）。US-02 の「ログアウト後は 401」は
  「トークン破棄後のリクエストにはトークンが付かないため 401」として満たす
- **sessions テーブルは削除**（マイグレーション be1c61339c9f・downgrade で復元可能）
- **CORS の許可オリジンは FRONTEND_ORIGIN（.env）で明示管理**（`*` にしない）

### 2026-07-05: スプリント10（最終）で確定した判断

- **E2E は Playwright・2シナリオに限定**（クリティカルパス + 認可）。
  単体テストと重複させない（テストピラミッドの配分: 63 + 25 + 2）
- **E2E の CI 組み込みは見送り**: DB + API + フロント + ブラウザの4層構築と
  フレーク対策は教材スコープ外と判断。ローカル実行手順を文書化し、
  発展課題として演習編に掲載
- **プロジェクト完了**: 全10スプリント・全18 PBI・US-01〜08 の全受け入れ条件を達成
  （M1: PR #1〜#4、M2: PR #5〜#10）
