# ユーザーストーリー

「〜として、〜したい、なぜなら〜」形式で、TODO アプリの利用者視点の要求をまとめる。

本プロジェクトは2部構成（第1部 API編 / 第2部 フロント編）のため、受け入れ条件も
**API編（curl / pytest で確認）** と **フロント編（ブラウザ / Vitest / Playwright で確認）**
に分けて書く。1つのストーリーは、両方の受け入れ条件を満たしてはじめて完了となる。

ステータスコードの原則（詳細は概念解説集で扱う）:

- **401 Unauthorized**: 未ログイン（認証の問題 =「あなたが誰か分からない」）
- **403 Forbidden**: 他人のリソースへの操作（認可の問題 =「誰かは分かるが権限がない」）
- **404 Not Found**: 存在しないリソース
- **422 Unprocessable Entity**: バリデーションエラー（Pydantic が自動生成）

## US-01: ユーザー登録

> 新規ユーザーとして、メールアドレスとパスワードでアカウントを登録したい。
> なぜなら、自分専用の TODO リストを持ちたいからだ。

**受け入れ条件（API編）**
- [ ] メールアドレスとパスワードで登録でき、`201` が返る
- [ ] パスワードは平文では保存されない（bcrypt によるハッシュ化）
- [ ] 登録済みメールアドレスでの再登録は `409` で拒否される
- [ ] メール形式が不正、またはパスワードが8文字未満なら `422` が返る

**受け入れ条件（フロント編）**
- [ ] 登録フォームから登録でき、完了後にログイン済み状態で一覧画面へ遷移する
- [ ] 入力エラー（形式不正・文字数不足・重複メール）が画面に表示される

## US-02: ログイン・ログアウト

> 登録済みユーザーとして、ログイン・ログアウトしたい。
> なぜなら、自分の TODO を安全に出し入れしたいからだ。

**受け入れ条件（API編）**
- [ ] 正しいメールアドレスとパスワードでログインでき、以降のリクエストが認証済みになる
      （第1部: cookie セッション。第2部後半: JWT に移行）
- [ ] 資格情報が誤っていると `401` が返る。メールアドレスの存在有無が推測できないよう、
      エラーメッセージは同一にする
- [ ] ログアウト後は認証が必要な API が `401` になる

**受け入れ条件（フロント編）**
- [ ] ログインフォームからログインでき、一覧画面へ遷移する
- [ ] 認証エラーが画面に表示される
- [ ] ログアウトするとログイン画面へ戻り、保護されたページに直接アクセスしても
      ログイン画面へリダイレクトされる

## US-03: TODO の作成

> ユーザーとして、タイトル・説明・期限日を指定して TODO を作成したい。
> なぜなら、やるべきことを忘れないよう記録したいからだ。

**受け入れ条件（API編）**
- [ ] title（必須・1〜100文字）、description（任意・最大1000文字）、
      due_date（任意・過去日も許可）で作成でき、`201` と作成結果が返る
- [ ] completed は指定しなければ `false` になる
- [ ] title 欠落・101文字の title・1001文字の description は `422`（境界値: 100文字/1000文字は成功）
- [ ] 未ログインでは `401`

**受け入れ条件（フロント編）**
- [ ] 作成フォームから TODO を作成でき、一覧に反映される
- [ ] バリデーションエラーがフィールド近くに表示される

## US-04: TODO の一覧表示

> ユーザーとして、自分の TODO を一覧で見たい。
> なぜなら、今やるべきことを一目で把握したいからだ。

**受け入れ条件（API編）**
- [ ] 自分の TODO だけが返る（他人のものは混ざらない）
- [ ] 既定は作成日時の降順・1ページ10件。`page` / `per_page`（上限100）で制御でき、
      レスポンスに総件数 `total` を含む
- [ ] `completed=true|false` で状態を、`q=キーワード` で title / description の
      部分一致を絞り込める（組み合わせ可）
- [ ] 未ログインでは `401`

**受け入れ条件（フロント編）**
- [ ] 一覧画面に title・期限日・完了状態が表示される
- [ ] ローディング中・エラー時・0件時にそれぞれ適切な表示になる
- [ ] 状態・キーワードで絞り込め、ページを移動できる

## US-05: TODO の詳細表示

> ユーザーとして、TODO の詳細（説明や期限日を含む全項目）を見たい。
> なぜなら、一覧では省略される情報を確認したいからだ。

**受け入れ条件（API編）**
- [ ] 自分の TODO の ID を指定すると全項目が返る（`200`）
- [ ] 存在しない ID は `404`、他人の TODO は `403`、未ログインは `401`

**受け入れ条件（フロント編）**
- [ ] 一覧から詳細画面へ遷移でき、全項目が表示される
- [ ] 存在しない ID の URL を直接開くとエラー表示（Not Found）になる

## US-06: TODO の編集

> ユーザーとして、TODO のタイトル・説明・完了状態・期限日を編集したい。
> なぜなら、状況の変化に合わせて記録を最新に保ちたいからだ。

**受け入れ条件（API編）**
- [ ] PATCH で指定した項目だけが更新され、指定しなかった項目は変わらない（部分更新）
- [ ] completed の true / false を切り替えられる
- [ ] バリデーションは作成時と同じ（違反は `422`）
- [ ] 存在しない ID は `404`、他人の TODO は `403`、未ログインは `401`

**受け入れ条件（フロント編）**
- [ ] 編集フォームで全項目を変更・保存でき、詳細・一覧に反映される
- [ ] 一覧または詳細から完了状態を切り替えられる

## US-07: TODO の削除

> ユーザーとして、不要になった TODO を削除したい。
> なぜなら、終わったこと・やらないことでリストを埋めたくないからだ。

**受け入れ条件（API編）**
- [ ] 自分の TODO を削除でき `204` が返る。再取得すると `404` になる
- [ ] 存在しない ID は `404`、他人の TODO は `403`、未ログインは `401`

**受け入れ条件（フロント編）**
- [ ] 削除前に確認ステップ（確認ダイアログ等）があり、キャンセルできる
- [ ] 削除後は一覧から消える

## US-08: 自分の TODO だけが扱えること（認可）

> ユーザーとして、自分の TODO が他人から見えたり操作されたりしないようにしたい。
> なぜなら、TODO には個人的な内容を書くからだ。

**受け入れ条件（API編）**
- [ ] 一覧に他人の TODO が一切含まれない
- [ ] 他人の TODO への詳細・編集・削除はすべて `403`
- [ ] 「本人は操作できる／他人は 403」を **ペアで** テストする
      （片側だけのテストでは認可の欠落を検出できないため）

**受け入れ条件（フロント編）**
- [ ] 通常の画面操作では他人の TODO に到達できない
- [ ] E2E で「ユーザーAの TODO がユーザーBに見えない」ことを確認する

## ストーリーとスプリントの対応

認証（US-01/02）より先に TODO の CRUD（S2〜S3）を作るため、S2〜S3 の時点では
TODO は所有者なしで動かし、S4 で users テーブルと `user_id` を **既存データの
マイグレーションつきで** 追加する。これは「稼働中テーブルの変更はデータ移行つきで」を
体験する意図的な設計である。

| ストーリー | API編（第1部） | フロント編（第2部） |
|---|---|---|
| US-01 登録 | S4 | S6 |
| US-02 ログイン・ログアウト | S4 | S6（S8 で JWT に移行） |
| US-03 作成 | S3 | S7 |
| US-04 一覧 | S2（認可は S4） | S6（S9 で TanStack Query に移行） |
| US-05 詳細 | S2（認可は S4） | S6 |
| US-06 編集 | S3 | S7 |
| US-07 削除 | S3 | S7 |
| US-08 認可 | S4 | S10（E2E で最終確認） |
