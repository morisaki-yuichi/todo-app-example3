# プロダクトロードマップ

## プロダクトゴール

CRUD 機能を持つマルチユーザー TODO アプリ（FastAPI + React）を、スクラム開発の流れに
沿って完成させる。同時に、**中級エンジニアが一人で写経・追体験できる教材ドキュメント一式**
を成果物として残す。

姉妹プロジェクト（Laravel + Plain HTML/CSS 版）と同じ題材をモダンな SPA 構成で作り直す
ことで、次の対比を学べる教材を目指す。

- **サーバレンダリング vs SPA**: 画面を誰が組み立てるか（サーバ or ブラウザ）
- **規約優先 vs 組み立て式**: フレームワークが決めてくれる世界 vs 自分で選んで積む世界

## 2部構成

| 部 | 内容 | 完了の定義（マイルストーン） |
|---|---|---|
| **第1部 API編** | バックエンド（FastAPI）だけを curl / 自動テストで完結させる | CRUD + 認証・認可を備えた API が pytest 全緑・CI 通過で動作する（M1） |
| **第2部 フロント編** | React で UI を載せ、API と結合する | ブラウザから全機能が使え、Vitest / Playwright が全緑・教材ドキュメント完成（M2） |

第1部でバックエンドを完結させるのは、**各コミットが「その層で完結」する**ように積むため。
API が curl とテストで検証済みなら、第2部で問題が起きたとき「フロント側の問題」と
切り分けやすくなる（データの通り道を外側から1段ずつ確認する調査の型にもつながる）。

## 段階導入の設計意図

この教材では、あえて「素朴なやり方 → 課題を体感 → ライブラリ / 仕組みで解決」の順で進める。

1. **素の fetch + useEffect → TanStack Query**（S6 → S9）:
   ローディング・エラー・再取得を手書きする大変さを先に体験してから、
   TanStack Query が何を肩代わりしてくれるかを学ぶ
2. **cookie セッション + Vite プロキシ → JWT + CORS**（S4 → S8）:
   まず「同一オリジン扱いなら CORS は要らない」状態で動かし、
   プロキシを外して CORS エラーを観察してから、別オリジン構成（JWT + CORS）へ移行する

## 横断ルール（全スプリント共通）

- 各スプリントで「プランニング → 実装 → レビュー → レトロスペクティブ」を必ず回す。
  レトロで出した Try は次スプリントで実際に使い、結果も記録する
- 各スプリントに **「わざと失敗を見る実験」を1回以上** 入れる（予想を先に書いてから実行）
- 「1コミット = 1つの意味のある変更」。動かない中途半端な状態でコミットしない
- スプリント終了ごとに開発トレースガイド（dev-walkthrough.md）と概念解説集（concepts.md）を追記する
- ドキュメントもコードと同様にコミットし、コミット前に他言語文字の混入がないか校正する
