## Keep（続けること）

- **内側から外側へ（設定 → モデル → ルータ → テスト）の編集順序**。
  各段を単体で検証でき、途中のエラーの疑い先が常に直前の1段に絞れた
- **autogenerate の生成物を必ず目視レビューしてから upgrade**。
  「提案として扱う」姿勢は、今後カラム変更で autogenerate の限界を踏むときの保険になる
- **実 Postgres でのテスト**。SQLite 代用にしなかったことで、CI にも同じ構成
  （サービスコンテナ）を素直に持ち込めた

## Problem（困ったこと・うまくいかなかったこと）

- 手動確認（curl）の段階で、日本語クエリの URL エンコード問題と、シードスクリプトの
  `-m` 実行問題という「アプリ外の落とし穴」を2つ踏んだ。どちらも実害はないが、
  自動テストだけでは見つからない種類の問題だった（手動確認の価値の再確認でもある）
- スプリントの分量が多め（コミット9個）。S3 は CRUD 3本だが、バリデーションの
  実験も入るため、詰め込みすぎに注意が必要

## Try（次のスプリントで試すこと）

1. **T-3: 手動確認のコマンド集を「写経者がコピペで再現できる形」でレビュー記録に残す**
   （今回の curl 一式は形になった。S3 では POST/PATCH/DELETE の `-d` つき例を同様に）
2. **T-4: スプリント開始時にタスクの「重い/軽い」を見積もり、重いタスクを先頭に置く**
   （分量超過の兆候を早めに検知して、スコープを削る判断を前倒しする）

→ この Try を S3 で実際に使い、結果を S3 のレトロに記録する。

<summary>回答例・ヒント（クリックで展開）</summary>

- マイグレーションは「スキーマ変更のコード化」。autogenerate は DB とモデルの差分から
  提案を作るので、DB が最新でないと拒否される（current と heads の見比べで診断）
- 依存性注入（Depends + dependency_overrides）のおかげで、テストは本物の DB 接続を
  テスト専用 DB に丸ごと差し替えられる。ルータ内で接続を直接作っていたら不可能だった
- 同じ接続コードでも「ホストから見た DB（localhost:5433）」と「コンテナから見た DB
  （db:5432）」は別の住所。接続エラーは「どこから見た住所か」をまず確認する
- ページネーションの並び順には一意なタイブレーカー（id）を入れる。同時刻データの
  順序不定は「たまに落ちるテスト」の定番原因
