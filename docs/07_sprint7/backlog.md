## 設計上の主な選択（理由つき）

- **TodoForm は作成・編集で1つの部品を共用**: フィールド構成・バリデーション表示は
  同一で、違いは「初期値」と「送信時の処理」だけ。props（`initial` / `onSubmit`）で
  差し込む
- **クライアント側で文字数制限（maxLength）を付けない**: バリデーションの信頼の
  源泉はサーバ（Pydantic）。クライアント制限は UX の補助にすぎず、二重管理すると
  ずれる。教材としては「サーバの 422 を画面に正しく届ける」経路を主役にする
  （title の必須だけは HTML の required で早期に止める）
- **422 の表示は `loc` でフィールドに割り付ける**: `detail[].loc` は
  `["body", "title"]` の形（S3 実験③）。`loc[1]` をキーにフィールド別エラーに変換する
- **削除の確認は `window.confirm`**: 誤操作防止の要件はダイアログの見た目ではなく
  「1クッション」があること。標準 API なら実装もテスト（モック）も最小。
  カスタムモーダルへの置き換えは演習編の題材にする
- **更新後は再取得（refetch）**: 完了トグル後の一覧は、手元の配列を書き換えるのでは
  なくサーバから取り直す。「サーバが真実」を保つ素朴で確実な方式
  （キャッシュ更新の洗練は S9 の TanStack Query で扱う）

## スコープ外（やらないこと）

- 楽観的更新（optimistic update）・キャッシュ管理（S9）
- カスタム確認モーダル（演習編）
- JWT / CORS（S8）
