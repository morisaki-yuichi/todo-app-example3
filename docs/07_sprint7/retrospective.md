## Keep（続けること）

- **api 層の関数を1スプリント先行して整備する**（T-10 の継続効果）。画面側は
  「関数を呼んで結果を描く」ことに集中でき、テストもモック1箇所で済んでいる
- **404 を「エラー」でなく専用状態として扱う**（not-found を状態の1値に）。
  エラーメッセージの流用より、ユーザーへの説明が具体的になる
- **実験の結果が予想と違ったときの記録**（実験⑥⑧）。「警告・挙動は環境依存であり、
  CI で回る検査を網にする」という方針が2つの実験から帰納できた

## Problem（困ったこと・うまくいかなかったこと）

- **テストが4本落ちた状態で一度コミットしてしまった**（Link 追加による Router
  ラッパー不足）。新しいテストだけ実行して安心し、全体を回さずにコミットした
  のが原因。push 前だったので amend で回復したが、「1コミット = 動く状態」の
  ルールを破った瞬間だった
- コミット前の確認手順が「暗黙」になっている（test → lint → build を毎回手で
  打っている）。抜けが今回の事故につながった

## Try（次のスプリントで試すこと）

1. **T-13: コミット前チェックを1コマンドにする**（`npm run check` =
   lint + test + build を直列実行。バックエンドにも同種の合成コマンドを検討）
2. **T-14: 実験は「予想 → 観察 → 検出網の確認」の3点セットで書く**
   （実験⑦⑧で定着した形式を、S8 の CORS 実験でも明示的に使う）

→ この Try を S8 で実際に使い、結果を S8 のレトロに記録する。
