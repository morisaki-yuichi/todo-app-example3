## 実験

### 実験⑪: mutation の invalidateQueries を外すとどうなるか

- **予想（実行前に記録）**: トグルの PATCH は成功する（サーバは更新済み）が、
  一覧のキャッシュが古いままなので画面は変わらない。エラーも警告も出ない。
  トグルのテスト（「一覧を取り直す」）が落ちて検出するはず
- **実際**: 予想どおり。テストは該当の1本だけが落ち、他6本は通った
  （= 問題の局所化にも成功）
- **検出網**: 「操作 → 再取得」の振る舞いテスト。実験⑦（依存配列）と同じ構図で、
  **「変えたのに画面が変わらない」系のバグはテストでしか機械検出できない**
  （lint はキャッシュの意味論まで見えない）
- **学び**: キャッシュを持つことの代償は「古くなったと宣言する責任」
  （invalidateQueries）。さらに実ブラウザでは refetchOnWindowFocus（既定 on）が
  ウィンドウ切替時に取り直すため、この手のバグは「たまに直る」ように見えて
  再現条件の特定が難しくなる——既定の自動再取得は安全網でもあり、バグ隠しでもある

## トラブル記録

### mutationFn に api 関数を直接渡すと第2引数が混入

- **症状**: `mutationFn: createTodo` とした TodoNew のテストが
  「expected to be called with...」で失敗。実際の呼び出しに
  `{client: QueryClient, meta, mutationKey}` という**第2引数**が付いていた
- **調査**: 失敗メッセージの Received 差分をそのまま読む（調査の型）。
  TanStack Query は mutationFn に変数 + コンテキストを渡す仕様で、
  関数参照を直接渡すと余分な引数まで届く
- **解決**: `mutationFn: (values) => createTodo(values)` と包み、
  「値だけを渡す」契約を明示
- **学び**: ライブラリのコールバックに自前関数の参照を直接渡すときは、
  引数の契約が一致しているかを確認する（JS は余分な引数を黙って受けるため、
  テストのモック検証で初めて表面化した——モックが仕様の食い違いの検出器になった例）
