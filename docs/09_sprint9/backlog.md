## スプリントゴール

**「データ取得・更新が TanStack Query に置き換わり、手書きしていた仕組み
（3状態の管理・cancelled フラグ・reloadKey 再取得）がライブラリの責務になる。
画面の見た目と振る舞いは変えない」状態を作る。**

## なぜ導入するのか（このスプリントの意味）

S6〜S7 で「素の fetch + useEffect」の大変さを体験済み:
①ローディング/エラー/成功の3状態を毎画面手書き、②StrictMode 対応の cancelled フラグ、
③更新後の再取得（reloadKey カウンタ）、④画面を離れて戻るたびに全部取り直し（キャッシュなし）。
TanStack Query は「**サーバ状態の管理**」を専門にするライブラリで、この4つを
queryKey という1つの概念で肩代わりする。**手作りの痛みを知ってから使う**のが本教材の設計。

## 設計上の主な選択（理由つき）

- **queryKey = 「このデータは何か」の住所**: 一覧は `['todos', {page, completed, q}]`、
  詳細は `['todo', id]`。条件が変われば別キー = 別キャッシュ。
  依存配列（S6 実験⑦）で手書きしていた「条件が変わったら取り直す」が、
  キーの一致・不一致に置き換わる
- **更新後は `invalidateQueries`**: reloadKey カウンタの正統な後継。
  「このキーのキャッシュはもう古い」と宣言すれば、表示中の画面は自動で再取得される
- **`retry: false` を既定にする**: TanStack Query の既定（3回リトライ）は本番向き
  だが、教材・テストでは「失敗がすぐ見える」方が良い。QueryClient の既定で無効化し、
  理由をコードコメントに残す
- **Home はあえて素の fetch のまま残す**: 「手作り版がどうだったか」を
  いつでも見に行ける対比サンプルとして保存する（コメントで明示）
- **api 層（src/api/）は無変更**: S6 の T-10 で切った層がそのまま queryFn /
  mutationFn になる。2スプリント連続で「層の切り方」が回収される
