## Keep（続けること）

- **「同じテストが通る」ことをリファクタリングの完了条件にする**。今回の移行は
  テストのアサーションをほぼ変えずに完走した。S6 でテストを「振る舞い」に対して
  書いていた（実装詳細でなく）ことが効いた
- **手書き → ライブラリの順**。素の fetch の痛み（3状態・cancelled・reloadKey）を
  知っているから、TanStack Query の各機能が「何の代わりか」を1対1で説明できた。
  この対応表（dev-walkthrough）は教材の中でも要になる
- **api 層の温存**。2スプリント連続（S8 認証・S9 データ取得）で、
  src/api/ を差し替えポイントとして活用できた

  型レベルでは通ってしまっていた）

<summary>回答例・ヒント（クリックで展開）</summary>

- サーバ状態は「本当の持ち主がサーバにいる」データ。クライアント状態（フォーム入力）と
  性質が違うので、専用の道具（キャッシュ + 再取得）で扱うと毎画面の手書きが消える
- queryKey はキャッシュの住所。「条件が変わったら取り直す」は依存配列から
  キーの一致・不一致に置き換わった（同型の書き漏らしバグも住所の書き漏らしとして残る）
- キャッシュの代償は「古くなったと宣言する責任」（invalidateQueries）。
  忘れてもエラーは出ない。振る舞いテストだけが機械的な検出網になる
- リファクタリングの成否は「同じテストが通るか」で判定できる。そのためには
  テストを実装詳細でなく振る舞いに対して書いておく必要がある
