## Keep（続けること）

- **制約は宣言としてスキーマに置き、ルータは薄く保つ**。バリデーション・422 応答・
  `/docs` の表示が1つの宣言から揃う気持ちよさは、このスタックの核心
- **「送らない」と「null」の区別を設計段階で決めてから実装**。
  PATCH の典型バグ（未送信項目の null 上書き）を最初から回避できた
- **境界値をテスト名に出す**（100/101、1000/1001）。仕様がテストから読める

## Problem（困ったこと・うまくいかなかったこと）

- PATCH と DELETE のルータ実装を勢いで同時に書いてしまい、「1コミット=1つの意味」の
  ために DELETE をいったん外して2コミットに分け直す手戻りが出た。
  コミット計画を先に決めていても、実装の勢いで崩れることがある
- 認証がまだ無いため、US-03/06/07 の受け入れ条件のうち 401/403 が未達のまま
  「API編としては完了」と判定している。この「部分完了」の状態管理は
  対応マップ（dev-walkthrough）で追えるようにしたが、S4 で確実に回収する必要がある

## Try（次のスプリントで試すこと）

1. **T-5: コミット単位で実装する**（次のコミットに入らないコードは書き始めない）。
   「書いてから分割」より「分割してから書く」ほうが手戻りがない
2. **T-6: スプリント開始時に「前スプリントからの持ち越し受け入れ条件」を
   バックログに明記する**（S4 なら「US-03〜07 の 401/403」。回収漏れを防ぐ）

→ この Try を S4 で実際に使い、結果を S4 のレトロに記録する。

- 入力の制約はルータの if 文ではなくスキーマの宣言で書く。すると 422 の自動生成・
  `/docs` への反映・テストの観点（境界値）まで一直線につながる

- PATCH の本質は「送らない = 変更しない」と「null = 消す」の区別。
  `exclude_unset` を知らずに実装すると、送っていない項目を null で潰すバグになる
- 422 の detail は loc → type → msg → input の順で読む。loc があるから
  フロントは「どの入力欄にエラーを出すか」を機械的に決められる
