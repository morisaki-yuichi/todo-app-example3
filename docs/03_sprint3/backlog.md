## 設計上の主な選択（理由つき）

- **入力は `TodoCreate` / `TodoUpdate` という専用スキーマで受ける**:
  `Todo`（table=True）で直接受けない。理由は2つ。
  ①クライアントに `id` や `created_at` を指定させないため（受け口に無い項目は入れようがない）、
  ② **table=True の SQLModel は Pydantic バリデーションを実行しない**という
  仕様上の落とし穴があるため（バリデーションは非テーブルのスキーマ側で行う）
- **編集は PATCH（部分更新）**: 「送られた項目だけ更新」を `exclude_unset` で実現する。
  「項目を送らない（変更しない）」と「null を送る（消す）」を区別できるのが PUT 全置換との違い
- **title は null にできない**: description の null は「説明を消す」として許可する一方、
  title の null は field_validator で 422 にする（DB の NOT NULL を 500 で踏む前に
  入り口で止める）
- **削除は 204 No Content**: 消えたものについて返す本文はない。「2回目の DELETE は 404」
  という素朴な仕様とする
