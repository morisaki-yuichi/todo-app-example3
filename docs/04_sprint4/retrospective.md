## Keep（続けること）

- **失敗を先に見る → 直す構成**（実験④）。「正解の3段階マイグレーション」だけを
  写経するより、naive 版の失敗を見てから直すほうが、なぜ3段階なのかが体に入る
- **ペアテスト**。実験⑤で「認可チェックを外すとペアテストの片側が落ちる」ことを
  実証できた。認可のテスト形式として今後も必須とする
- **401 の判定を deps に集約し、403 を共通関数（get_owned_todo）に寄せる**。
  エンドポイントが増えても認可の書き忘れが構造的に起きにくい

## Problem（困ったこと・うまくいかなかったこと）

- **コンテナ内 venv への依存同期を忘れて curl ハング**に時間を使った
  （調査自体は `docker compose logs api` で一発だったが、ハングの発生から
  ログを見るまでに間があった）。「依存を足したら api を再起動」をルール化した
- テストが 55 本になり、実行時間も伸び始めた（bcrypt を使う認証テストが重い）。
  conftest では「セッション行の直接挿入」でログインを高速化したが、
  今後も「本物を通すテスト」と「速いテスト」の使い分けの意識が必要
- スプリント分量は今回も多め。第2部では「1スプリント = 画面1〜2枚」を目安に
  もう少し薄く切る

## Try（次のスプリント = 第2部開幕で試すこと）

## 振り返り課題（写経者向け）

**「このスプリントで学んだことを、自分の言葉で3つ要約してみよう。」**

<details>
<summary>回答例・ヒント（クリックで展開）</summary>

- 401 は「誰か分からない」、403 は「誰かは分かるが権限がない」。判定の場所も分ける
  （401 は認証依存に集約、403 は所有チェックに集約）と書き忘れが構造的に減る
- データが入ったテーブルへの NOT NULL 追加は「nullable で追加 → 埋める → 締める」。
  autogenerate はデータを知らないので、データ移行は必ず人間が書き、down まで検証する
- 認可の欠落は例外もエラーログも出ない。「本人はできる／他人は 403」のペアテストが
  唯一の機械的な検出器になる
- パスワードは「わざと遅い」bcrypt でハッシュ化し、ソルトで「同じパスワード = 同じハッシュ」
  を防ぐ。ログイン失敗の理由は攻撃者に教えない（メール列挙対策）

</details>

## 第1部 API編を終えて（中間ふりかえり）

環境構築 → Read → CUD → 認証・認可の4スプリントで、curl と pytest だけで完結する
マルチユーザー TODO API が完成した（M1 達成）。第2部は React でこの API に UI を載せる。
第1部で API が「テスト済みの既知の量」になっているため、第2部で問題が起きたときは
まずフロント側を疑える——これが2部構成にした狙いである。
