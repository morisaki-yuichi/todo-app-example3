## 設計上の主な選択（理由つき）

- **パスワードハッシュは bcrypt を直接使う**: 当初案の passlib は長期未メンテで
  最新 bcrypt と組み合わせ問題があるため、bcrypt ライブラリを直接使う
  （hashpw / checkpw の2関数で足り、概念の学習内容は変わらない）。QAログにも記録
- **セッションは DB 保存（sessions テーブル）+ httpOnly cookie**: cookie には
  ランダムなトークンだけを入れ、実体はサーバ側に持つ「ステートフル」方式。
  ログアウト = サーバ側の行削除で**即時無効化できる**のが特徴で、
  S8 で移行する JWT（ステートレス・即時失効が苦手）との対比の土台になる
- **ログイン失敗は理由を区別しない**: 「メール未登録」と「パスワード違い」を
  同一メッセージの 401 にする（ユーザー列挙攻撃への対策。仕様は QAログ #4）
- **既存 TODO の所有者は移行用ユーザーに割り当てる**: マイグレーション内で
  `legacy@example.com`（ログイン不可のハッシュ値）を作成し、既存行を割り当てて
  NOT NULL 化する。「稼働中テーブルの変更はデータ移行つきで」の実演

## スコープ外（やらないこと）

- JWT / CORS（S8 で cookie 方式と対比しながら移行）
- CSRF 対策の本格実装（概念解説では触れる。同一オリジン + SameSite=Lax の範囲で運用）
- パスワードリセット・メール確認・レート制限（教材スコープ外として明記）
- フロントエンド
