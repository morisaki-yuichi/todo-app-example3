## Keep（続けること）

- **api 層モックによる画面テスト**。fetch レベルより意図が読みやすく、
  「API が返す値 → 画面の振る舞い」に集中できる
- **lint を実験の検出網として扱う**。実験⑦で「テスト + lint」の2重検出を確認できた。
  警告ゼロ運用（S1 から継続）がここでも効いた
- **エラーメッセージのステータス分岐**（ApiError.status）。第1部で規律よく
  ステータスコードを設計したおかげで、フロントの分岐が素直に書けた

## Problem（困ったこと・うまくいかなかったこと）

- RTL の cleanup 漏れで5テストが落ち、調査に時間を使った（globals: false 構成の
  落とし穴。既知の人には常識だが、初見だと「テストの独立性」問題だと気づきにくい）
- Todos.tsx が 160 行まで育ち、1ファイルの密度が高くなってきた。
  Pager は切り出したが、一覧アイテムやフィルタ部も S7 でフォームが加わると
  分割が必要になりそう

## 振り返り課題（写経者向け）

**「このスプリントで学んだことを、自分の言葉で3つ要約してみよう。」**

<details>
<summary>回答例・ヒント（クリックで展開）</summary>

- useEffect の依存配列は「この値が変わったら再実行」の宣言。書き漏らすと
  エラーの出ない「画面が更新されない」バグになるが、lint とテストの2重の網で防げる
- SPA の認証は「起動時に /auth/me で cookie から復元」が基本形。復元中の
  「不明」状態を持たないと、ログイン済みなのにログイン画面へ飛ばす誤判定が起きる
- fetch を画面に直書きせず api 層を1枚かませると、テスト（モック）も将来の
  ライブラリ移行もその層で完結する
- 「単体では通るのに一括で落ちる」テストはテスト間の状態共有（DOM の残骸・
  DB の残データ）を疑う——バックエンドでもフロントでも同じ原則
